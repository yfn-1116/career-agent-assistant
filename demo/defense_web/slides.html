调度层：OrchestratorAgent

LLM 收到消息 → 判断意图："这是岗位分析任务" → 决定：先解析JD+索引知识库 → 再检索匹配 → 再分析 → 最后生成

主 Agent 怎么分派任务？

OrchestratorAgent 的 LLM 决策循环 — 以这个 JD 为例
1
LLM 判断 → "需要解析 JD 并索引知识库"
调 task_agent("解析JD+索引PDF和GitHub", [parse_jd, web_search]) → 生成 SubAgent-1
2
LLM 判断 → "需要检索匹配的经历" ⚡与 Step1 并行
调 task_agent("检索Agent开发相关经历", [retrieve_profile, github_repo]) → 生成 SubAgent-2
3
LLM 判断 → "需要分析匹配度" 🔗 依赖 Step2 结果
调 task_agent("分析JD和证据的匹配度", [analyze_match, grade_retrieval]) → 生成 SubAgent-3
4
LLM 判断 → "生成简历建议"
调 task_agent("生成简历建议", [generate_answer, check_faithfulness, write_report]) → 生成 SubAgent-4
5
LLM 判断 → "任务完成，回复用户"
不再调 Tool，直接输出最终答案给用户
主 Agent 不干活，只做决策。干活的是子 Agent。
8/10

子 Agent 怎么执行、怎么回传？

以 SubAgent-1（感知·解析专家）为例

📄 SubAgent-1：感知·解析专家

Role: 信息采集专家 Session: 独立上下文 max_steps: 8
System Prompt: "你是信息采集专家，解析所有输入并索引到知识库。"

Permissions: parse_jd web_search

执行过程：
① 调 parse_jd(JD文本) → {job_title, hard_skills: [Python, Java, LLM, Agent, 工具调用, 知识库管理, 对话系统...], direction: "agent"}
② 调 FileLoader(PDF) → pypdf提取 → TextChunker分块(800字) → SHA256 ID
③ 写入 chunks.jsonl → 3个chunk索引完成
通信流程：
① 主Agent调 task_agent
task = "解析JD文本，索引PDF和GitHub到知识库"
allowed_tools = [parse_jd, web_search]
▼
② SubAgent-1 独立执行
自己的 System Prompt + 自己的上下文
不受其他 Agent 干扰
▼
③ 返回 SubAgentResult
{answer: "JD已解析, PDF索引3chunk", tools_called: ["parse_jd", "web_search"]}
子 Agent 也能再调 task_agent → 生成孙 Agent（递归）
终止条件：max_steps 硬限制

RAG：知识库的数据怎么进去、怎么查出来

写入路径（Perception→Memory） + 读取路径（Memory→Action）

📥 写入路径

1
PDF简历 → pypdf 提取文本
2
GitHub链接 → MCP(JSON-RPC) 调 npx 读 README
失败回退: urllib 直接 GET raw
3
TextChunker 滑动窗口分块(chunk=800, overlap=100)
4
SHA256 document_id → 追加 chunks.jsonl

📤 读取路径（查询: "Python Agent LLM 工具调用 知识库管理"）

1
jieba 中文分词 → token 集合
2
双路并行粗召回 (各取 top 80)
BM25 关键词 ⊕ Qwen Embedding 语义(1024维)
互补：BM25保准确，Embedding保召回
3
RRF 融合 (k=60, → top 30)
只关心排序位置，不关心分数绝对值，零参数
4
CrossEncoder 精排 (→ top 5)
bge-reranker-base 279M · query+chunk 拼一起过Transformer
5
FaithfulnessChecker (阈值 0.75)
6
EvidenceGate 四级约束
✅ implemented→可写 ⚠️ designed→需确认 ❌ planned→禁止
⚡ API 不可用 → 轻量回退：BM25 + 五维规则打分，<50ms，零网络依赖
