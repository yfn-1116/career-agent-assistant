# 模型供应商选择

## 用途

本文档记录第一阶段模型 provider 的边界和抽象策略。

## 候选方向

- DeepSeek：可作为主要模型 provider，用于中文求职语境和成本控制。
- OpenAI 或其他 provider：可作为后续替换或对照。
- Mock / rule-based 输出：可用于早期测试和 demo 兜底，降低模型不可用风险。

## 决策

代码实现不应绑定单一模型。第一阶段可以先通过接口或配置抽象模型调用，并允许 mock / rule-based 输出用于测试和演示兜底。

## 原因

1. 模型可用性会受网络、额度、学校服务器环境影响。
2. 不同任务对模型质量和成本要求不同。
3. demo 当天需要离线或低依赖兜底。
4. 抽象 provider 可以避免后续大面积改代码。

## 第一阶段边界

- 不提交真实 API Key。
- 不在文档补强任务中引入依赖。
- 不把 prompt 和 provider 强绑定。

## 后续维护规则

- provider 接口和配置必须由明确任务设计。
- 模型输出必须接受事实一致性检查。
