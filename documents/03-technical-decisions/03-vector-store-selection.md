# 向量库选择

## 用途

本文档比较简单内存检索、Chroma 和 FAISS，并确定第一阶段向量存储策略。

## 候选方案

### 简单内存检索

优点：实现快、依赖少、适合 demo 和小规模样例。缺点：持久化和大规模检索能力弱。

### Chroma

优点：本地持久化体验较好，适合 RAG 原型。缺点：需要新增依赖，环境差异可能影响学校服务器复现。

### FAISS

优点：检索性能强，适合向量规模增长。缺点：安装和环境兼容性可能更复杂，不适合文档阶段提前引入。

## 决策

第一阶段先抽象 `VectorStore` 接口，具体实现可以先用内存或轻量本地方案，后续再接 Chroma / FAISS。

## 接口设计影响

`VectorStore` 接口应至少考虑：

- 写入 chunks。
- 查询相似 chunks。
- 返回 score 和 metadata。
- 清空或重建索引。
- 后续持久化扩展。

## 后续维护规则

- `RAG-004` 由 Codex 负责，因为接口会影响后续 Chroma / FAISS / memory store 切换。
- 不允许 loader 或 Agent 直接依赖具体向量库。
