# 验收标准

## 用途

本文档定义第一阶段需求、设计、实现和 demo 的验收标准。

## 功能验收

1. 输入一个示例 JD，系统能解析岗位关键词、技术栈、职责和候选人要求。
2. 系统能从用户资料中检索出相关经历，并保留证据来源。
3. 系统能说明匹配原因，包括强匹配点、弱匹配点和缺口。
4. 系统能生成一段基于真实经历的简历项目描述。
5. 系统能生成一段基于真实经历的 HR / mentor 沟通话术。
6. 生成内容不应明显编造经历、指标、奖项或公司名称。
7. 当资料不足时，系统应提示缺少证据，而不是强行生成确定结论。

## 工程验收

1. 能在本地运行 CLI demo。
2. 能输出 Markdown 结果文件。
3. 能通过 GitHub 拉取到学校服务器并复现 demo。
4. Streamlit 展示如果进入第一阶段，应只调用已有 workflow，不承载核心业务逻辑。

## 文档验收

1. 任务实现前有对应 task 或 plan。
2. 每个任务声明允许修改和禁止修改的文件。
3. 完成后更新 `documents/97-journal.md` 和 `documents/99-project-planning.md`。

## 当前状态

当前处于文档补强阶段，尚未开始功能验收。

## 后续维护规则

- 每个实现任务必须指向至少一个验收标准。
- 验收标准应尽量可通过命令、样例或人工检查验证。
