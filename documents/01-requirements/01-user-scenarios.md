# 用户场景

## 用途

本文档记录第一阶段必须服务的实习求职场景，用于指导 RAG、Agent 和 demo 的设计。

## 场景一：判断岗位是否适合投递

用户粘贴一个 Agent 实习、AI 应用开发或软件开发实习岗位 JD，希望系统解析岗位要求，并基于自己的项目经历、技能材料和实习经历判断是否适合投递。

- 输入：岗位 JD Markdown 或纯文本。
- 期望输出：岗位关键词、能力要求、匹配经历、匹配原因、短板提醒。
- 边界：系统只给出辅助判断，不代替用户做最终投递决策。

## 场景二：根据 JD 改写项目经历

用户希望把已有课程项目、GitHub 项目或实习项目改写成更贴合岗位 JD 的简历项目描述。

- 输入：JD、用户项目资料、目标岗位方向。
- 期望输出：基于真实经历的项目描述、技术栈突出点、岗位相关表达。
- 边界：不得编造用户没有做过的功能、指标或成果。

## 场景三：生成 HR / mentor 沟通话术

用户希望基于真实背景生成一段简洁、可信、适合发送给 HR、mentor 或内推人的沟通话术。

- 输入：JD、匹配分析、用户经历证据。
- 期望输出：中文沟通话术、可替换占位、语气建议。
- 边界：不得承诺不存在的经历，不输出夸大事实的表达。

## 场景四：避免模型编造经历

用户希望系统严格基于资料库内容输出，不能把模型猜测写成用户经历。

- 输入：用户资料库和生成任务。
- 期望输出：每个关键结论能追溯到资料证据，无法确认时明确提示。
- 边界：推测和建议必须与事实经历分开。

## 场景五：学校服务器展示

用户希望本地开发完成后，将代码推送到 GitHub，再在学校服务器拉取并展示 demo。

- 输入：GitHub 仓库、示例用户资料、示例 JD。
- 期望输出：可复现的 CLI 或 Streamlit demo。
- 边界：第一阶段不要求完整生产部署和用户账号系统。

## 当前状态

上述场景是第一阶段设计和验收的基础，后续任务不得绕开这些场景扩展无关功能。

## 后续维护规则

- 新场景必须说明输入、输出和边界。
- 场景若影响核心模块，需要同步更新设计文档和任务卡。
