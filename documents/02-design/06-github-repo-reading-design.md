# GitHub 仓库读取设计

## 用途

本文档定义第一阶段如何处理用户 GitHub 仓库信息，以及后续扩展 `GitHubRepoAgent` 的边界。

## 第一阶段策略

第一阶段不直接读取完整 GitHub 源码，而是将 GitHub 仓库整理为本地 Markdown 摘要，字段包括：

- 仓库名称。
- 仓库链接。
- README 摘要。
- 技术栈。
- 项目亮点。
- 适配岗位方向。
- 用户实际负责内容。
- 可用于简历的证据片段。

## 为什么不直接读取完整源码

1. 避免一次性塞入大量源码导致上下文污染。
2. 降低第一阶段开发复杂度。
3. 先验证求职场景下的资料检索与匹配能力。
4. 后续可以作为 Tool Calling / MCP 扩展点。

## 后续 GitHubRepoAgent 扩展范围

后续需要时，再按需读取：

- `README`
- `docs/`
- 目录结构
- 核心代码文件
- 测试文件
- 配置文件

## 第一阶段输入示例方向

一个仓库摘要 Markdown 应能回答：

- 这个项目解决什么问题？
- 使用了哪些技术？
- 用户负责哪些模块？
- 哪些内容能证明岗位要求中的能力？
- 这个项目适合匹配哪些岗位方向？

## 当前状态

当前只定义策略，不实现 GitHub API、爬取或源码分析能力。

## 后续维护规则

- 真实 GitHub 读取能力必须单独开任务卡。
- 不在没有授权的情况下读取私有仓库。
- GitHub 摘要应优先服务求职匹配，而不是完整代码理解。
